## 1. 지난 데모(원본) vs 오늘 업그레이드

### 지난 데모버전 (`Desktop/KMTP-MVP`)
- **단일 환자용 5단계 위저드**: 국가·진료과 → 가격잠금 견적 → 예약 슬롯 → 에스크로 결제 → 신뢰·후기

- **로그인 없음**, 역할 구분 없음
- Vercel에 **프론트만** 배포

## 2. 이번 데모의 장점

- **풀스택 실데이터 흐름**: 단계 진행·알림·계정이 백엔드에 실제 저장되어, 단순 화면 시연을 넘어 동작을 보여줌.
- **역할별 격리**가 명확해 B2B2C **신뢰 인프라** 모델(환자·에이전트·병원·운영자)을 설득력 있게 전달.
- **No-Surprise/가격잠금/Trust 5축** 등 핵심 가치가 화면으로 구체화.
- **카톡 스타일 메시지**로 실사용 경험을 직관적으로 시연(비용 0).
- **단계별 git 커밋 + 백업 지점**으로 안전하게 진행, 언제든 롤백 가능.
- **무료 인프라**(Vercel/Render 무료)로 비용 없이 라이브 공개.

## 3. 이번 데모의 단점 / 한계

- **보안 미흡**: 비밀번호 평문, 세션/토큰 없음(새로고침 시 로그아웃), 일부 역할 검증을 클라이언트에 의존.
- **데이터 영속성 없음**: Render 무료 SQLite가 재시작 시 초기화 → 가입 계정·진행 기록 소실(시드만 복원).
- **콜드 스타트**: 백엔드 15분 미사용 시 슬립 → 첫 접속 지연(30~60초).
- **mock 의존**: 결제·카톡·웨어러블·후기 모두 가짜 데이터.
- **테스트 부재**: 자동 테스트(단위/E2E) 없음.
- **반응형/접근성** 정밀 점검 미실시.

## 4. 앞으로 추가로 구현해야 할 점 (Roadmap)

### A. 가이드의 "나중에" 항목
- **지표 화면**: 저장된 시각으로 픽업 평균 대기시간·노쇼율 계산·표시.
- **씨어스 회복데이터·만족도**: 빈 테이블(`recovery_monitoring`, `satisfaction`) 채우고 화면 추가(1주/1개월/3개월).
- **영구 인터넷 운영**: 영구 DB 연결(아래 B 참고).

### B. 기반 강화 (실서비스화 전 필수)
- **영구 DB**: Render Postgres(무료) 또는 디스크 → 가입/진행 데이터 보존.
- **진짜 인증**: 비번 해시(bcrypt) + JWT/세션 + 새로고침 유지 + 로그아웃.
- **권한 서버검증 강화**: 모든 보호 API에 역할 확인.
- **테스트**: 백엔드 pytest, 프론트 Playwright(역할별 흐름).

### C. 미완 / 다듬기
- **여정 완료 → 신뢰 자동 이동** (이전에 논의했으나 미구현).
- **모바일 반응형·접근성** 점검.
- **에러/로딩 UX** 통일.

## 5. 추가하면 좋을 기능 (Feature Ideas)

- **관리자 기능 확대**: 계정 수정/삭제, 환자–에이전트–병원 **배정** 관리, 대시보드.
- **실제 알림 연동**: 카카오 알림톡 / 이메일 / 웹푸시(비용·심사 고려).
- **다국어(i18n)**: 중/영/아랍/일/베트남/러시아어 — 타깃 국가가 7개로 확대됨.
- **결제 PG 연동**(에스크로 실거래), 환불·분쟁 처리 흐름.
- **실시간 업데이트**: 폴링(현재 5초) → WebSocket/SSE.
- **파일/사진**: 의료기록·여권·영수증 업로드, 후기 사진.
- **에이전트 성과·정산 화면**(수수료는 환자 비노출 유지).
- **병원 슬롯 실연동**(현재 mock 시간표) + 캘린더.
- **웨어러블(씨어스 ECG) 실데이터** 회복 모니터링.
- **검색/필터**: 진료과·국가·병원 비교.
