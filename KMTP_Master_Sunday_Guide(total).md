# KMTP 통합 마스터 작업 지시서 (일요일)

> 클로드 코드 입력창에 **위에서부터 하나씩** 복붙하세요.
> 두 가지(① 실시간 여정+백엔드 ② 환자 견적·예약·신뢰 강화)를 한 앱에 모두 담는 통합본입니다.

---

## 역할별로 무엇이 보이나 (전체 그림)

이 앱은 **Patient / agent / hospital** 세 역할로 로그인하고, 로그인한 역할에 따라 다른 화면을 봅니다.

| 역할 | 보이는 것 |
|---|---|
| **patient** | 견적·가격잠금·회복시설·신뢰점수·No-Surprise 확인 **+ 내 여정 실시간 추적** |
| **agent** (유치업자) | 담당 환자들의 여정 관리 + 픽업 진행 (견적 상세·수수료는 안 보임) |
| **hospital** | 예약 환자의 도착 상태·진료 일정 (어느 에이전트를 통했는지·수수료는 안 보임) |

> **그래서 "환자 견적·예약·신뢰 강화"는 전부 patient 로그인 영역에 들어갑니다.** 예외는 인트로 화면과 이해관계자 맵(About)뿐 — 이건 역할 무관 공통 화면입니다.
> 화이트라벨 원칙: 환자 화면 어디에도 KMTP가 직접 서비스한다는 게 드러나지 않습니다.

**기술 구성:** Next.js 프론트 + Python FastAPI / SQLite 백엔드 (로컬 시연까지가 일요일 목표, 인터넷 배포는 다음에)

---

## 데이터 모델 (백엔드가 저장할 것)

**지금 만들 테이블:** patients, agents, hospitals, appointments(예약시간·실제 진료 시작·종료), transfers(픽업 예정·기사 도착·탑승 완료·기사 전화·차량번호), journey_events(단계·타임스탬프), notifications(수신자·채널·내용·읽음)

**빈 자리만 만들 테이블 (나중에 값 채움):** recovery_monitoring(씨어스), satisfaction(1주/1개월/3개월)

> 노쇼율·픽업 대기시간은 따로 저장하지 않고 '예정시간 vs 실제시간' 차이로 나중에 계산합니다. 일요일엔 시간을 빠짐없이 기록하는 데 집중.

# Part 1 — 골격: 3역할 + 백엔드 + 여정 (핵심 기반)

## 작업 2 — 3역할 로그인 골격 (기존 견적 흐름을 patient 영역으로)

```
지금 앱은 환자가 견적받고 예약하는 5단계 흐름이야. 여기에 Patient/agent/hospital 3개 역할 로그인을 추가하려고 해.

- 세 개의 mock 계정(patient/agent/hospital)으로 로그인하는 간단한 화면을 만들어줘 (복잡한 인증 말고)
- 로그인한 역할에 따라 다른 화면으로 가게 해줘:
  - patient: 기존 견적·예약·신뢰 흐름 (+ 뒤에서 여정 추적을 추가할 자리)
  - agent: 담당 환자 관리 (뒤에서 채움)
  - hospital: 예약 환자 관리 (뒤에서 채움)
- 기존 견적·예약·신뢰 화면(StepQuote, StepSlot, StepEscrow, StepTrust 등)은 patient 로그인 영역으로 옮겨줘
- 화이트라벨: 환자 화면 어디에도 KMTP가 직접 서비스한다는 게 드러나지 않게

기존 틸 디자인 유지. 끝나면 확인법 알려줘.
```

## 작업 3 — Python 백엔드 + 데이터 모델 + API

```
이제 Python으로 백엔드를 만들어서 환자 여정 데이터를 DB에 저장할 거야. 나는 초보니까 단계별로.

1. Python이 깔려 있는지 확인하고 없으면 설치 도와줘
2. 기존 Next.js 프로젝트는 두고, 같은 프로젝트 안에 'backend' 폴더로 FastAPI + SQLite 백엔드를 만들어줘 (가상환경 포함)
3. 테이블: patients(식별코드·국적·진료과·담당에이전트·병원), agents(에이전시명·연락처), hospitals(병원명·담당자연락처), appointments(환자·병원·예약시간·실제 진료 시작/종료·상태), transfers(유형 공항→숙소/숙소→병원/병원→숙소·기사 이름/전화/차량번호·픽업 예정/기사 도착/탑승 완료 시각·상태), journey_events(환자·단계·발생시각·메모), notifications(환자·수신자역할·채널·내용·발송시각·읽음)
4. 나중에 채울 빈 테이블도: recovery_monitoring(환자·측정시각·지표·값·출처seers), satisfaction(환자·시점1주/1개월/3개월·점수·코멘트)
5. API: 여정 이벤트 기록/조회, 픽업·예약 등록/조회, 알림 저장/조회. 프론트(localhost:3000)에서 백엔드(localhost:8000) 호출되게 CORS 설정. mock 환자 2~3명·에이전트 1명·병원 1곳 샘플 데이터 넣어줘

다 되면 백엔드 실행법(uvicorn 등)과 확인법 알려줘.
```

## 작업 4 — 여정 타임라인 + 역할별 격리

```
환자 여정을 타임라인으로 보여주는 화면을 만들어줘. 단계: 현지출발 → 공항도착 → 공항 픽업 → 숙소/회복스테이 체크인 → 병원 방문 → 수술·시술 → 회복 → 재진 → 출국. 각 단계에 완료/진행중/예정 표시.

역할별 격리:
- '현재 어느 단계인지'는 patient/agent/hospital 모두에게 보임
- patient: 자기 일정·픽업 연락처·병원 위치/시간
- agent: 담당 환자들의 현재 단계 목록과 처리할 일 (견적·수수료 안 보임)
- hospital: 예약 환자의 도착 상태·진료 일정 (에이전트·수수료 안 보임)

데이터는 작업 3 백엔드 API에서 가져와. 기존 디자인 유지. 확인법 알려줘.
```

## 작업 5 — 단계 진행 → 저장 + 인앱 알림 + 픽업 정보

```
여정 타임라인에서 단계를 진행시킬 수 있게 해줘.
- '공항 픽업 완료' 같은 버튼을 누르면 백엔드에 그 이벤트가 시각과 함께 저장되고, 관련 역할에게 화면 안 알림(팝업/알림목록)이 뜨게
- 알림 규칙: 공항 픽업 완료 → 환자·에이전트 + 병원 예약시간·장소 안내 / 병원행 탑승 완료 → 환자·에이전트·병원 / 진료 종료(숙소 복귀) → 환자·에이전트
- 알림 기록은 notifications 테이블에도 저장
- 실제 카톡/문자가 아니라 '앱 안에서 뜨는 알림'으로 시뮬레이션 (나중에 실제 발송으로 교체)
- 픽업 단계 카드에는 기사 이름·전화번호·차량번호·픽업 예정시간 표시 (mock). 환자 화면엔 "기사에게 전화" 버튼, agent 화면엔 담당 환자 픽업 진행 현황

기존 디자인 유지. 확인법 알려줘.
```

- 신뢰 3모듈 카드: 가격잠금 / 슬롯동기화 / 검증후기

기존 디자인 유지. 확인법 알려줘.

## 작업 7 — 견적: 가격잠금 3종 + 환율 안내 + 회복시설 자동 매칭

```
patient의 견적 화면을 강화해줘.
- 진료과에 따라 가격잠금 카드가 3종으로: Full Lock(뷰티·치과·안과·검진, 총액 고정) / Range Lock(암·척추·관절, 최소~최대 범위+변동룰) / No Lock(응급·이식, 상담 견적)
- 환율 안내 한 줄: "표시 가격은 자국 통화 기준, 입국 시점 환율 변동은 ±3% 이내 자동 흡수"
- 진료과마다 표준 회복기간을 정해두고(갑상선 14박·피부성형 7박·정형외과 21박·검진 2박 등), 예약 단계에서 그 기간에 맞춰 회복스테이 객실(스탠다드/디럭스/패밀리)이 자동으로 함께 추천·예약되게
데이터는 lib/data.ts와 백엔드에 추가. 기존 디자인 유지. 확인법 알려줘.
```

## 작업 8 — No-Surprise 증명 화면

```
patient 흐름 마지막 근처에 "예약 때 잠근 가격 = 입국 후 청구 가격" 일치를 눈으로 증명하는 화면을 추가해줘.
- 왼쪽: 예약 시점에 잠근 견적 / 오른쪽: 입국 후 실제 청구서 (같은 금액·항목)
- 큰 표시: "가격 일치 — 추가 청구 없음 ✓"
- "환율 변동 ±3% 이내 자동 흡수" 한 줄, Range Lock은 "사전 합의된 변동 룰에 따름"
데모니까 항상 일치하게. 기존 디자인 유지. 확인법 알려줘.
```

## 작업 9 — Trust Score 5축 + 검증후기 + 회복 경과

```
patient의 신뢰 화면을 강화해줘.
- Trust Score를 5축 종합점수로: ①가격잠금 준수율 ②슬롯등록 성실도 ③결과지표 응답률 ④환자 만족도 ⑤합병증율(낮을수록 좋음). 막대그래프나 점수로
- 문구: "이 점수는 광고비로 살 수 없고 실제 거래 데이터로만 산출됩니다" / "외부 후기는 표시하지 않고, 결제·시술이 확인된 검증 후기만 보여줍니다". 검증 후기엔 '검증됨 ✓' 마크
- 그 아래 "회복 경과 추적": 시술 후 1주/1개월/3개월 만족도·합병증·재방문 추적 + "웨어러블(ECG)로 모니터링" 한 줄 (mock: 1주 양호/1개월 양호)
데이터는 lib/data.ts. 기존 디자인 유지. 확인법 알려줘.
```

## 작업 10 — 국가·환자 페르소나 확대

```
국가를 중국·몽골·중동 + 일본·베트남·러시아/CIS·미국 교포로 확대해줘. 각 국가에 대표 환자 페르소나(나이·성별·관심 진료과·체류기간)와 인기 진료과를 mock으로. (일본: 미용·검진 / 베트남: 피부·성형 단기 / 러시아·CIS: 종합검진+심혈관 / 미국 교포: 치과+검진). 국가를 고르면 그 나라 인기 진료과가 먼저 보이게.
데이터는 lib/data.ts. 기존 디자인 유지. 확인법 알려줘.
```
